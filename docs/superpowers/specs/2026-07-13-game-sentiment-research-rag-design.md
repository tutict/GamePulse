# GamePulse 游戏舆情研究 RAG 设计

日期：2026-07-13

## 目标

GamePulse 从通用的本地评论数据工具收敛为报告优先的游戏舆情研究应用。用户只需手动填写游戏名称，并可选填写本次关注问题；本地 Agent 自动发现主流公开来源、采集评论与讨论、纠正无关结果、建立本地证据库，并生成带引用的风评分析报告。

首版继续支持 Windows 与 Android。两端数据独立保存在本机，不恢复中心服务器，也不提供后台同步。

## 非目标

- 不把项目管理、文件导入、全文搜索、通用问答或独立证据页作为一级功能。
- 不承诺覆盖整个互联网，也不承诺绕过登录、验证码、付费墙、地区限制或强反爬机制。
- 不修改评论原文。Agent 只能排除、标记、降权或重新分类证据。
- 不做定时后台舆情监控、自动通知或持续运行的云端采集任务。
- 不引入向量 embedding 或端侧大模型。首版继续使用 SQLite FTS5 与远程模型。

## 产品流程

### 1. 发起研究

应用首页为极简研究入口，只包含：

- 游戏名称，必填，由用户手动填写。
- 重点关注问题，可选。留空时生成标准风评报告；填写时重点研究指定问题。
- 开始研究按钮。
- 最近研究列表。

默认研究最近 90 天，并与可获取的历史口碑进行比较。导航只保留“开始研究”“历史报告”和“设置”。

### 2. Agent 自动研究

研究依次执行：

1. 解析游戏身份，排除同名内容。
2. 发现主流公开来源。
3. 采集评论与讨论。
4. 规范化、去重并统一时间与语言元数据。
5. 对相关性、来源质量和身份置信度评分。
6. 建立本地 FTS5 证据库。
7. 生成结构化风评报告并校验引用。

研究进度页只展示对用户有意义的阶段、已覆盖来源、有效证据数量和可恢复错误，不暴露底层队列或调试信息。

### 3. 查看报告

报告采用结论优先的层级：

1. 一句话总体判断。
2. 正面、负面与中性口碑概览，以及相对历史口碑的变化。
3. 玩家最关注的主要主题排序。
4. 核心优点、核心问题和主要争议。
5. 每个判断对应的证据数量、来源范围与代表评论。
6. 研究覆盖范围、失败来源和可信度说明。

网络采集得到的是非随机公开样本。情绪比例和主题数量用于描述本次样本，不表达对全部玩家的统计推断；界面必须明确说明这一限制。

### 4. 后续操作

- 基于证据追问：只检索当前研究的本地证据，不在问答期间扩展样本。
- 更新研究：由用户手动触发，生成新的报告版本，并展示与上一版本的口碑变化。
- 纠正来源：在来源与证据抽屉中排除错误来源或评论，记录排除原因，然后重新生成报告。
- 导入数据：保留为高级补充入口，用于用户拥有的 CSV、JSON、NDJSON 或 `.gamepulse` 数据，不进入默认主流程。

## 前端结构

Windows 与 Android 共享研究工作台的产品语言和组件边界：

- `ResearchStart`：研究输入、凭据前置检查和最近研究。
- `ResearchProgress`：阶段进度、来源覆盖、取消和可恢复错误。
- `SentimentReport`：报告摘要、主题、趋势、引用与追问。
- `EvidenceDrawer`：来源审计、原文查看、排除与重新生成。
- `ResearchHistory`：报告版本、更新时间和口碑变化。
- `Settings`：模型、搜索服务和采集能力配置。

“项目”继续作为内部存储隔离单位存在，但在 UI 中统一称为“研究”，不要求用户理解项目包或数据库结构。

响应式规则：

- 手机使用顶部品牌栏和不超过三个一级入口的底部导航。
- 桌面使用窄侧栏或顶部导航，但内容主轴始终是研究与报告，不展示工具集合式仪表盘。
- 报告在窄屏按结论、主题、证据纵向排列；宽屏可将概览与来源说明并排，但不把报告改成高密度控制台。

## 采集架构

共享层定义 `ResearchCollector` contract，输入研究目标和时间范围，输出统一的来源记录、原始证据和采集诊断。

### Windows

- 外部搜索服务发现 Steam、TapTap、Bilibili、贴吧、Reddit、公开媒体和论坛候选页面。
- 平台公开接口优先于页面解析。
- Electron Chromium 采集需要渲染、滚动或动态加载的公开页面。
- 采集运行在 Electron 主进程或受控采集窗口中，renderer 不直接获得任意网络或文件权限。

### Android

- 外部搜索服务发现候选来源。
- 使用公开平台接口和 Capacitor 原生网络能力采集可访问内容。
- Android 首版不提供 Electron Chromium 等价能力；必须登录或强反爬的来源记为不可访问，并进入覆盖说明。

## 本地研究流水线

研究编排器将采集与分析拆分为可测试阶段：

- `GameIdentityResolver`：根据游戏名、页面标题、发行商、平台标识和别名判断目标身份。置信度不足时暂停并要求用户选择候选游戏。
- `SourceDiscovery`：生成候选来源并记录发现方式。
- `EvidenceNormalizer`：保留原文，规范化用于哈希和检索的副本。
- `EvidenceDeduplicator`：按内容哈希、规范化 URL 和近似重复信号去重。
- `EvidenceQualityScorer`：计算游戏相关性、时间有效性、来源质量和身份置信度。
- `ReportGenerator`：从筛选后的证据生成结构化报告。
- `CitationValidator`：拒绝没有证据引用的主要判断，并验证引用属于当前研究版本。

Agent 自动纠错的含义是排除无关目标、降低低置信度证据权重、合并重复内容和标记来源限制。任何纠错都不能覆盖原始评论字段。

## 数据模型

SQLite 增加或扩展以下概念：

- `Research`：游戏名称、关注问题、时间范围、状态、目标身份和创建时间。
- `ResearchRun`：一次采集运行的阶段、进度、诊断、开始时间、结束时间和取消状态。
- `Source`：平台、URL、标题、发现方式、访问状态和采集时间。
- `Evidence`：来源引用、原文、规范化内容、作者公开标识、发布时间、语言、哈希和质量评分。
- `Exclusion`：被 Agent 或用户排除的对象、原因、操作者和时间。
- `ReportVersion`：结构化结论、主题、口碑概览、引用、覆盖说明和基于的研究运行。

现有 `Project`、评论、标签和报告数据通过兼容层映射到研究模型。旧项目仍可打开，项目包导入导出保留，但入口降级到历史研究和高级操作中。

## RAG 与报告约束

- 报告生成使用经过质量筛选的研究证据，并采用结构化输出格式。
- 每个主要结论至少绑定一条引用；证据不足时必须明确写出不确定性。
- 报告追问继续使用本地 FTS5 检索、去重、重排和上下文裁剪。
- 默认追问最多使用 8 条证据、12,000 字符上下文、同一来源最多 2 条。
- 更新研究创建新的 `ReportVersion`，不覆盖旧报告。
- 用户排除证据后，只重新运行筛选、报告生成和引用校验，不重复联网采集。

## 错误与恢复

- 游戏身份不明确：暂停研究并展示候选游戏，用户确认后继续。
- 搜索服务或模型凭据缺失：开始研究前引导设置，并保留已填写内容。
- 部分来源失败：研究继续，报告显示成功、失败和排除的来源。
- 有效证据不足：不生成确定性风评结论，展示覆盖情况和扩大范围建议。
- 报告格式无效：自动重试；仍失败时保留证据，并允许单独重新生成报告。
- 用户取消：终止后续采集，将运行标记为已取消，不生成报告。
- 应用重启：未完成研究恢复为可继续或可重试状态，不自动重新联网。

## 隐私与合规

- 评论原文、来源 URL、报告和研究历史默认保存在本机 SQLite。
- 搜索与模型服务只接收完成当前步骤所需的最小输入。
- API Key 继续使用 Electron `safeStorage` 或 Android Keystore-backed adapter 保存。
- 项目包继续排除 API Key、令牌、模型缓存、设备路径和采集会话信息。
- UI 明确列出实际覆盖来源，并允许用户打开公开原始页面。
- 采集器遵守可访问性边界，不实现验证码规避或登录凭据窃取。

## 测试与验收

### 自动测试

- 研究编排器各阶段的单元测试。
- Windows 与 Android 采集适配器的 contract tests。
- 游戏身份歧义、去重、排除和质量评分测试。
- 报告结构解析和引用完整性测试。
- 部分来源失败、无证据、取消、重试和应用恢复测试。
- 研究旧数据兼容和报告版本迁移测试。
- React 组件的空状态、进度、错误、报告和证据排除交互测试。
- Playwright 覆盖桌面、平板和手机视口，检查导航、文字溢出、报告展开和对话框焦点。

### 外网验证

稳定 CI 使用固定搜索结果和采集样本，不依赖真实外网。发布前执行手动联网冒烟测试，至少覆盖一个静态公开来源、一个 Electron 动态来源、一次部分失败和一次来源排除后重生成。

### 完成标准

- 用户可以仅填写游戏名和可选问题发起研究。
- Agent 能展示来源发现、采集、清洗和报告生成进度。
- 报告提供总体判断、口碑变化、主题、优缺点、争议和可检查引用。
- 错误来源可被排除，重新生成后不再参与结论。
- 报告追问不联网，并只引用当前研究证据。
- Windows 与 Android 的核心研究流程一致，平台能力差异有明确提示。
- 旧项目数据与 `.gamepulse` 文件仍可读取，不丢失现有本地数据。

## 实施边界

本次改造优先完成前端信息架构、研究领域类型、研究编排 contract、固定样本驱动的 Agent 流程和报告生成闭环。真实平台采集器按来源逐个接入，不以“全网覆盖”作为单次发布的完成条件。
